Architecture Overview

SiteIQ is built on a modular architecture using pytest as the test execution framework, Flask for the web interface, and specialized utilities for advanced security testing.

Web Application

Flask-based UI with real-time streaming

Test Modules

pytest-based security, SEO, GEO, LLM tests

Utilities

OAST, Toxicity, Adversarial, Orchestrator

Payloads

SQLi, XSS, LLM attack vectors

Reports

JSON reports with findings

Test Framework pytest Test discovery, execution, fixtures, markers

Utilities

Advanced utility modules that provide specialized capabilities for security testing.

OAST Client

Out-of-Band Application Security Testing for blind vulnerability detection.

utils/oast_client.py 
from utils.oast_client import OASTClient, OASTConfig

# Initialize client with Interactsh server
config = OASTConfig(server="oast.live")
client = OASTClient(config)

# Generate unique callback URL for a test
callback_url = client.generate_callback("ssrf_test_001")
# Returns: http://abc123.oast.live

# Check if callback was triggered (blind vuln detected)
if client.has_interaction("ssrf_test_001"):
    print("Blind SSRF vulnerability confirmed!")

Capabilities: DNS callbacks, HTTP callbacks, Interactsh integration, local callback server fallback

Toxicity Scorer

ML-based detection of toxic, harmful, or inappropriate LLM outputs.

utils/toxicity_scorer.py 
from utils.toxicity_scorer import ToxicityScorer, ScorerBackend

# Initialize with Perspective API
scorer = ToxicityScorer(
    backend=ScorerBackend.PERSPECTIVE,
    api_key="your_perspective_api_key",
    threshold=0.7
)

# Score LLM response for toxicity
result = scorer.score("Some LLM response text...")
print(f"Toxicity: {result.score}, Toxic: {result.is_toxic}")
print(f"Categories: {result.categories}")  # threat, insult, profanity, etc.

Backends: Perspective API (Google), Toxic-BERT (local), Signature-based (fallback)

Adversarial Optimizer

GCG-style adversarial suffix optimization to bypass LLM safety filters.

utils/adversarial_optimizer.py 
from utils.adversarial_optimizer import AdversarialOptimizer, OptimizationMode

optimizer = AdversarialOptimizer()

# Define query function for target LLM
def query_llm(prompt):
    return requests.post(llm_endpoint, json={"message": prompt}).json()

# Run adversarial optimization
result = optimizer.optimize(
    prompt="Tell me how to...",
    target_goal="Get LLM to comply",
    query_fn=query_llm,
    mode=OptimizationMode.GENETIC  # or STATIC, RANDOM_SEARCH
)

print(f"Best suffix: {result.best_suffix}")
print(f"Success: {result.success}")

Modes: Static (known suffixes), Random Search, Genetic Algorithm

Red-Team Orchestrator

Automated multi-turn attack conversations using LLM-as-Attacker pattern.

utils/orchestrator.py 
from utils.orchestrator import RedTeamOrchestrator, OrchestratorConfig, AttackStrategy

config = OrchestratorConfig(
    strategy=AttackStrategy.JAILBREAK,
    max_turns=10,
    attack_goal="Bypass safety guidelines"
)

orchestrator = RedTeamOrchestrator(
    config=config,
    attacker_query_fn=attacker_llm_query,  # Your attacker LLM
    target_query_fn=target_llm_query       # Target being tested
)

# Run automated attack conversation
result = orchestrator.run()
print(f"Success: {result.success}, Turns: {result.total_turns}")
print(orchestrator.get_conversation_log())

Strategies: Jailbreak, Extraction, Manipulation, Escalation

Detailed Flow
  1. Web Interface: User enters target URL and selects tests via dashboard
  2. API Endpoint: POST /api/scan receives config, validates input, creates test run
  3. Process Spawn: Flask spawns pytest subprocess with appropriate markers and CLI options
  4. Test Discovery: pytest discovers tests matching selected markers (e.g., -m security)
  5. Fixture Injection: conftest.py injects target_url, session, and utility fixtures
  6. Test Execution: Each test class sends payloads and analyzes responses
  7. SSE Streaming: Test output is streamed via Server-Sent Events to browser
  8. Report Generation: Results saved to reports/ as timestamped JSON

CLI Usage

SiteIQ can be run directly via pytest CLI for automation and CI/CD integration.

Basic Usage
# Run all security tests
pytest --target-url=https://example.com -m security -v

# Run specific test category
pytest --target-url=https://example.com -m sql_injection -v

# Run LLM security tests
pytest --llm-endpoint=https://api.example.com/chat -m llm -v

# Run SEO and GEO tests
pytest --target-url=https://example.com -m "seo or geo" -v
Advanced Options
# With OAST for blind vulnerability detection
pytest --target-url=https://example.com -m ssrf \
    --oast-server=oast.live -v

# With custom wordlist for forced browsing
pytest --target-url=https://example.com -m forced_browse \
    --wordlist=/path/to/wordlist.txt -v

# LLM tests with toxicity scoring
pytest --llm-endpoint=https://api.example.com/chat -m llm \
    --toxicity-scorer=perspective \
    --perspective-api-key=YOUR_KEY \
    --toxicity-threshold=0.7 -v

# LLM tests with adversarial optimization
pytest --llm-endpoint=https://api.example.com/chat -m llm_adversarial \
    --adversarial-mode=genetic -v

# Red-team orchestration
pytest --llm-endpoint=https://target.example.com/chat -m llm_orchestration \
    --attacker-llm=https://attacker.example.com/chat \
    --max-attack-turns=15 -v
Available CLI Options
Option Description
--target-url Target website URL for testing
--llm-endpoint LLM API endpoint URL
--oast-server OAST server for blind detection
--wordlist Custom wordlist for brute-forcing
--toxicity-scorer Toxicity backend (auto/perspective/toxic_bert/signature)
--toxicity-threshold Toxicity score threshold (0.0-1.0)
--perspective-api-key Google Perspective API key
--adversarial-mode Adversarial optimization mode (static/random/genetic)
--attacker-llm Attacker LLM endpoint for orchestration
--max-attack-turns Max turns for orchestrated attacks

Extending SiteIQ

Adding New Tests
  1. Create test file in tests/ directory (e.g., test_new_vuln.py)
  2. Add pytest marker in conftest.py
  3. Add test category to TEST_CATEGORIES in webapp/app.py
  4. Create payload file in payloads/ if needed
# tests/test_new_vuln.py
import pytest

class TestNewVulnerability:
    """Tests for new vulnerability type."""

    @pytest.mark.new_vuln
    @pytest.mark.security
    def test_basic_detection(self, target_url, session):
        """Test basic vulnerability detection."""
        response = session.get(f"{target_url}/endpoint")
        # Your test logic here
        assert "vulnerable_pattern" not in response.text
Adding New Payloads
# payloads/new_payloads.py

# Static payload list
NEW_VULN_PAYLOADS = [
    "payload1",
    "payload2",
    "payload3",
]

# Dynamic payload generator
def generate_context_payloads(context: str) -> list:
    """Generate payloads based on context."""
    return [f"{context}{p}" for p in NEW_VULN_PAYLOADS]
Test Categories: Remember that Security tests and SEO/GEO tests are mutually exclusive. LLM tests are also separate. Design your tests to fit within these boundaries.
